# Prompt 01 — ProjectKnowledgeBuilder

> **使用方式**：将 `extracted_decisions.md` 和 `code_summary.md` 的内容分别填入下方
> `<session_summary>` 和 `<code_summary>` 标签内，整体发送给 Claude（推荐 claude.ai，
> 需要足够大的 Context 窗口）。将输出保存为 `project_knowledge_graph.md`。

---

你是一个技术架构分析师。我会给你两份文档：

1. **Session 摘要**（`extracted_decisions.md`）：工程师与 Claude Code 的对话片段，反映了开发过程中的决策和讨论
2. **代码摘要**（`code_summary.md`）：项目的目录结构、依赖栈、提交历史和核心模块入口

请在**同一次分析**中交叉印证两份文档，构建项目知识图谱。

关键要求：
- 找到"session 里说要做的"与"代码里有没有实现"之间的张力，这是后续面试题的最高价值来源
- 置信度标注要严格：宁可标中/低，不要过度拔高
- 决策描述要具体（写模块名、技术名称），不要写"优化了性能"这种空话

---

## 输出格式

### 1. 技术栈与架构模式

列出项目使用的核心技术栈，以及整体架构模式（前后端分离/微服务/Monorepo/等）。
对每项技术，简要说明：在 session 里是否有选型讨论？代码里是否有对应实现？

### 2. 关键架构决策清单

列出至少 5 条关键架构决策，每条包含：

- **决策描述**：用一句话概括（必须包含具体技术/模块名）
- **背景**：为什么需要做这个决策？当时有什么约束条件？
- **取舍**：选择了 A 而不是 B，原因是什么？
- **置信度**：
  - 🟢 高：session 有明确讨论 + 代码有对应实现（两者都能印证）
  - 🟡 中：仅 session 提及但代码未能确认，或仅从代码结构推断
  - 🔴 低：纯推断，无直接证据，不建议在面试中主动提及

### 3. 交叉印证发现

指出以下情况（如有）：

- **Gap（说了没做）**：Session 里明确提到要做，但代码里没有对应实现的功能/方案
- **孤岛（做了没说）**：代码里出现的模块，在 session 里从未讨论过
- **方案变更**：Session 里有明显的思路转变痕迹（最终选择与最初想法不同）

### 4. 技术域分布

将决策按以下分类汇总（一条可属于多个分类）：

| 技术域 | 涉及决策 |
|--------|---------|
| 架构层（模块设计/服务拆分/依赖管理） | |
| 数据层（状态管理/持久化/缓存） | |
| Agent 层（LLM 调用/工具调用/记忆管理） | |
| 工程化层（构建/测试/CI） | |
| 性能层（并发/缓存/优化） | |
